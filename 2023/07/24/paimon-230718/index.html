Paimon-[卷I]-核心概念

01 总览

Streaming data lake platform with high-speed data ingestion, changelog tracking and efficient real-time analytics.

Paimon 是一个具有 高速数据摄取、变更日志跟踪 和 高效实时分析 功能的 流数据湖平台

官网、文档、GitHub、贡献指南、Roadmap

Paimon核心支持Flink的数据读取，同时也支持Hive、Spark、Trino等计算引擎的读取

1.1 架构

1.2 基本概念 Basic Concepts

  • Snapshot快照: 快照是对表再某一瞬间的 状态记录，因此可以让用户进行Time-traveling(时间旅行)；
  • Partition分区: 和Hive类似，Paimon将数据按照分区来切分数据块；
    如果定义了主键，分区键必须是主键的子集。
  • Bucket桶: 对于 非分区表、已分区表 进行 二次数据切分。用户可以通过提供 bucket-key 选项来指定bucket列
    Bucket 是针对 读/写的最小存储单元，因此bucket的数量限制了最大的处理并行性。但是，这个数字不应该太大，因为它将导致大量小文件和低读取性能。一般情况下，建议每个bucket中的数据大小为1GB左右
  • Consistency Guarantees一致性保证: 写表时，Paimon通过 两阶段提交协议 自动将一批记录提交出去（每次提交时最多产生两次snapshot）

02 核心

1.3 文件布局

  1. Snapshot Files快照文件: 快照文件均存储在上图的 snapshot 文件夹中，这些文件是 JSON 格式的，其中包含了 元数据信息 和 快照的变更记录列表
  2. Manifest Files清单文件: 清单文件均存储在上图的 manifest 文件夹中， 每个 manifest-list 中由1个以上的 manifest-file 组成，每个 manifset-file 包含了 LSM数据文件 和 changelog files
  3. Data Files数据文件: 数据文件由 partitions 和 buckets 所组织，每个桶目录包含了一个 LSM tree 以及它的 changelog files

1.3.1 简述 LSM-Trees

LSM 树将文件以多个 sorted runs序运行 组成。sorted runs 由多个 data files 组成，每个sorted runs 都是唯一的。

查询操作

不同的 sorted runs 中可能包含相同主键的 data files，所以 在查询LSM树的时候，必须按照用户指定的合并引擎规则搭配时间戳合并相同主键的sorted runs集合

写入操作

新纪录将首先在内存中缓冲，当缓冲区已满时，将对内存中的所有记录进行排序并刷写到磁盘中。

合并操作

当越来越多的数据写入LSM树，刷写出来的 sorted runs 将会堆积，为了处理因为大量 sorted runs 带来的查询性能低下(甚至是OOM情形)，Paimon使用了 Compaction 操作来合并小单位的 sorted runs。

类似于 Rocksdb 的 universal compation[^1]，Paimon 的Compaction策略也是在”读”/“写”之间进行性能的抉择，因为合并动作对于数据块来说是一个敏感的锁式操作，对于CPU和磁盘I/O影响较大。

快照过期

无论如何在SQL语义的层面增删数据，只要快照还未过期，都不会导致数据的真正删除。因为 Paimon 提供了时间旅行的功能用于回溯早期的快照版本。

具体和快照过期相关的参数如下：

  • snapshot.time-retained：快照的保留时间（默认1hour）
  • snapshot.num-retained.min：快照数目的阈值下限（默认10个）≥
  • snapshot.num-retained.max：快照数目的阈值上限（默认Integer.MAX_VALUE）≤

当快照数目小于 snapshot.num-retained.min 时，没有快照会被过期（即使达到了 snapshot.time-retained）
当快照数目唱过 snapshot.num-retained.min 后，按照 snapshot.time-retained 和 snapshot.num-retained.max 来控制过期策略。

1.5 主键表

Changelog表 是创建表时的默认表类型。用户可以在表中插入、更新或删除记录。
主键是每条记录唯一的一组列。Paimon会强制排序数据，这意味着在每个bucket中，数据是被排列好的，因此用户可以通过在主键上添加过滤条件来实现高性能。

1.5.1 合并引擎

Paimon Sink 接收到上游一批数据且其中包含重复组件数据时，会对那部分数据进行合并merge，以确保数据的唯一，用户可以通过指定 merge-engine 参数来定义合并策略。

  • 去重合并引擎deduplicate默认：将最近接触到的数据(全字段)替换之前的旧数据；
  • 部分更新引擎partial-update：将最近接触到的数据(部分字段)替换之前的旧数据；

    可以看做是除了主键外的字段全按照 COALESCE(column, null) 处理

  • 聚集合并引擎aggregation：按提供的聚合函数规则，将最近接触到的数据(部分字段)替换之前的旧数据。

    目前支持 sum、min/max、last_value/last_non_null_value、listagg、bool_and/bool_or
    并且当前仅有 sum 原生支持 回撤retraction，其他四个函数必须通过设置 'fields.${field_name}.ignore-retract'='true' 从而支持回撤。

  • 首行合并引擎first-row：保持首个数据。

流式查询时，partial-update引擎 & first-row引擎 必须和 lookup型 OR full-compaction型 的 Changelog 制造器一起使用！

partial-update引擎无法接受DELETE操作的信息，因为该行为无法被定义。用户可以通过配置 partial-update.ignore-delete=DELETE来忽略DELETE操作

first-row引擎仅支持插入顺序，所以无法为其指定 squence.field 参数，此外它也不接受 DELETE操作 和 UPDATE_BEFORE操作

在 FlinkSQL 中设置 table.exec.sink.upsert-materialize=NONE 总是会导致奇怪的行为，所以当明确摄入的数据是乱序的时候，请阅读并使用 序列化字段

1.5.2 Changelog 制造器

流式查询会不断生产出变更changes，我们可以通过指定 `` 这个表属性来进行制造器的切换。

  • None无：仅看得到合并后的数据结果，无法回溯旧的数据状态；不建议
  • Input(常规)输入型：将输入的记录直接刷写到独立的changelog文件中。建议
  • Lookup检索型：使用缓存辅助changelog的刷写。试验
  • Full Compaction全压缩：对于读延迟过高的场景可以使用该制造器，它分离了 数据刷写 和 日志生成 两个流程，加速查询，但是写延迟升高。不建议

1.6.1 压缩策略

可以控制压缩策略的配置：

  • write-only：default:false 只写模式下不跳过 数据压缩 和 快照过期 流程；
  • compaction.min.file-num：default:5 数据文件压缩触发阈值下限。
  • compaction.max.file-num：default:50 数据文件压缩触发阈值上限。
