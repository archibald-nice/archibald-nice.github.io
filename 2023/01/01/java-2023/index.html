01 常规通识

1.1 JVM

从JDK-1.8开始，原有的 方法区 已经被 元数据空间Metaspace 替换，和方法区不同 ，metaspace不再和堆共用连续的内存空间，而是使用本地内存，这意味着只要机器内存足够，那么不可能出现OOM！

1.2 常用集合

HashMap

哈希表由 数组 + (单向)链表 实现，链表长度超过8时转换为红黑树。

➡ 调用 put(k,v) 插入元素时：

  1. hashCode(k) 计算键的哈希值，按照数组长度进行负载按位与获得数组bucket桶的下标index；
  2. 判定bucket是否为空：
    1. bucket空，直接插入；
    2. bucket不为空，解决哈希冲突 => 遍历该桶中的链表（或红黑树，从 Java 8 开始引入），检查键对象是否已经存在：
      1. 键对象存在，新值替换旧值；
      2. 键对象不存在，尾插法插入表尾。

插入时如果 链表长度 超过 8TREEIFY_THRESHOLD，则链表转化为红黑树，提高查询效率；长度小于 6 时，又会转回链表
插入时如果 键值对对数目 超过 哈希表长度(default:16)与载因子(default:0.75)的乘积阈值capacity * loadfactor，则发生扩容resize(默认2倍扩容)
扩容操作会想创建一个2倍大小的新哈希表，然后将旧哈希表中的所有键值对重新插入到哈希表中

红黑树引入的原因：
早期HashMap使用的是基于拉链法的散列表来解决冲突。然而，当HashMap中的元素数量很大时，存在大量的散列冲突，导致链表长度过长，降低了查找、插入和删除操作的效率。
链表的时间复杂度为O(n)，而红黑树所有操作的**时间复杂度为O(logn)**，效率提升；
红黑树通过节点着色以及自旋功能提供了”自平衡“的高度控制，防止树的层级过深。

➡ 调用 get(k) 获取元素时：

  1. hashCode(k) 计算键的哈希值，获得index；
  2. 顺序遍历链表中每个”Node”，通过 equals 定位和k对应的v值
