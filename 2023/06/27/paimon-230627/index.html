Paimon-[番外]-Meetup_01期

01 Apache Paimon 0.4 及后续展望

1.1 湖存储上的难点

  1. 入湖痛点：

    1. 吞吐差，COW更新差、MOR查询差…
    2. 内部管理系统庞大：Compaction、清理历史小文件、清理过期分区…
    3. Schema Evolution[1] 湖存储同步上游对于数据列的增删改查变化，重启同步作业时无法准确拉取Schema

  2. 打宽表痛点：

    1. CDC Delete 怎么处理？
    2. 输入乱序 怎么办？
    3. Projection pushdown[2] 怎么处理？

  3. 流读痛点：

    1. 全量接增量读取 完整的流；
    2. Changelog 的读取 低成本/低时延；
    3. 支持Flink原生的 lookup join

1.2 深入 Apache Paimon 0.4

1.2.1 Paimon的读写&收益

Paimon 之所以能打，是因为引入了 LSM，但为什么要引入 LSM 呢？请看下图：

  1. 写入：Paimon 的文件操作是一个 湖存储 + LSM 的框架，其中 LSM 是一种 针对写友好 的格式，写入操作由 Flink Sink 维护在 Main Table 中，当 checkpoint 触发或者 Main Table 满了以后，就会对内存中的数据进行排序，然后 刷写flush 数据到 LSM 树的 L0 层中；
  2. 压缩：得益于 LSM 不断地 异步Minor Compaction，使得 L0 层的 局部数据 不断被合并出去，同时不会生成过多的层级，避免了 写放大 的问题；
  3. 清理：Flink 写入作业默认会清理过期的快照，有效的清理可以减少DFS上大量的文件，而且你还可以配置分区的清理策略；
  4. 读取：发生在新启动的作业中，您可以查询在文件系统上提交的快照。该查询将 有序合并Merge Read 多个 Level 的 LSM，这可能会有一些开销，但不会很慢。

总结：Paimon的特点 —— Append 写是大吞吐的写、局部压缩可以降低消耗、自动清理策略是全自动的、有序合并 不会太慢

同程旅行实践收益(对比Hudi)：

  • 资源节省：流计算资源节省30%、实时ODS存储资源节省40%
  • 性能提升：任务写入性能（全量+增量）提升3倍左右、部分查询性能提升7倍

1.2.2 核心功能概述

  • CDC入湖：
    • 表同步：表结构变更同步Schema Evolution + 主键 + 分区列
    • 整库同步：INCLUDING/EXCLUDING + 表名前后缀同步 + ⾃动跳过失败的表
  • 部分更新Partial update：下游表的更新可以定义每个字段的来源，从而实现不同字段识别不同上游而更新。同时支持了 Sequence Field 处理乱序问题。
  • 增强流读Stream-Read Enhancement：
    • 支持了原始数据的读取（changelog-producer=input）
    • 支持生成变更日志，Lookup 模式 cp 可见(1-3分钟时延)，Full-compaction 模式延迟可见 (3-10分钟时延)
    • 支持 Consumer-ID，支持流读无状态重启，也解决流读恢复时的 FileNotFound 问题
